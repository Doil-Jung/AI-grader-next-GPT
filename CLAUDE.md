# AI Grader vNext 병렬 개발 규칙

## 목적

- GPT 개발본과 Claude 개발본을 격리한 목적은 서로의 성과를 보지 못하게 비교하기 위한 것이 아니다.
- 목적은 두 AI가 같은 파일을 동시에 수정하여 코드, 프로젝트 데이터, Git 이력이 꼬이는 일을 막는 것이다.
- 다른 개발본과 기존 프로젝트를 읽고 참고하는 것은 허용하며, 필요하면 적극적으로 참고할 수 있다.

## 읽기 허용 범위

- 이 파일이 있는 자신의 개발 폴더 전체를 읽을 수 있다.
- 상위 `코딩 작업 폴더`와 그 아래의 다른 프로젝트를 읽고 참고할 수 있다.
- 특히 다음 폴더의 코드, 문서, 테스트, 프로젝트 자료와 결과를 읽고 비교할 수 있다.
  - 기존 앱 `ai_grader`
  - 공통 기준본 `ai_grader_next_baseline`
  - GPT 개발본 `ai_grader_next_gpt`
  - Claude 개발본 `ai_grader_next_claude`
  - 사용자가 참고하라고 지정한 다른 프로젝트
- 상대 개발본에서 유용한 구현을 발견하면 아이디어를 참고하거나 파일을 자신의 개발 폴더로 복사한 뒤 자기 개발본에 맞게 수정할 수 있다.

## 쓰기 경계

- 기본적으로 이 파일이 들어 있는 자신의 개발 폴더만 수정한다.
- 기존 앱, 공통 기준본, 상대방 개발본 및 상위 폴더의 파일을 생성·수정·삭제·이동·이름 변경하지 않는다.
- 상대방 폴더에서 명령을 실행하여 빌드 결과, 캐시, 로그 또는 임시 파일이 생기게 하지 않는다.
- 여러 개발본이 함께 쓰는 가변 폴더, 심볼릭 링크 또는 디렉터리 연결을 만들지 않는다.
- 자신의 `.git` 파일이 가리키는 외부 Git 저장소만 사용하며, 다른 개발본의 Git 저장소를 수정하지 않는다.
- 여러 개발본을 함께 변경해야 하는 작업은 먼저 사용자에게 알리고 명시적인 지시를 받은 뒤 진행한다.
- 사용자가 특정 폴더의 변경을 명시적으로 지시한 경우에만 이 기본 경계를 넘을 수 있다.

## 테스트 자료

- 자신의 `projects`와 `output`에 복사된 이전 프로젝트, PDF 및 결과 자료는 개발과 회귀 테스트에 사용할 수 있다.
- 원본 보존이 필요한 기존 자료는 가능한 한 덮어쓰지 않고, 생성 자료에는 구분 가능한 이름을 붙인다.
- 다른 개발본의 자료가 필요하면 읽기만 하거나 자신의 폴더로 복사하여 사용한다.
- API 키, 학생 자료 및 결과 데이터는 Git에 커밋하지 않는다.

## 경로와 생성 파일

- 소스, 문서, 테스트, 프로젝트 실행 데이터처럼 기기 간 이어서 작업해야 하는 파일은 자신의 원드라이브 개발 폴더에 둔다.
- `.git`, 패키지 설치 폴더, 브라우저 프로필, 빌드 산출물 및 대량 캐시는 원드라이브 밖의 개발본별 독립 위치를 사용한다.
- 코드에 특정 사용자명이나 절대 경로를 새로 고정하지 말고 상대 경로, 설정값 또는 환경 변수를 사용한다.

## 작업 원칙

- 상대 개발본을 참고하는 것은 자유지만 실제 변경은 반드시 자신의 개발본에서 수행한다.
- 작업을 시작하기 전에 현재 경로와 Git 작업 트리를 확인하여 자기 개발본이 맞는지 확인한다.
- 기존 사용자 변경 사항을 보존하고, 요청 범위와 무관한 파일을 정리하거나 되돌리지 않는다.
- 개발 과정에서는 EXE를 만들지 않으며, 사용자가 최종 테스트 후 빌드를 요청할 때만 빌드한다.
