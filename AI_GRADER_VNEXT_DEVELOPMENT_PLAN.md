## 1. 문서 목적

이 문서는 지금까지 개발된 AI 채점기의 기능과 사용자 논의를 하나로 정리하고, 다음 버전을 안정적으로 개발하기 위한 기준 문서다.

다음 질문에 일관된 답을 제공하는 것이 목적이다.

1. 이 앱이 학교 현장에서 해결해야 하는 핵심 문제는 무엇인가?
2. 기존 기능 중 무엇을 유지하고 어디로 이동시킬 것인가?
3. 보고서·수행평가, 대회·탐구 심사, 정기고사 서술형을 어떤 구조로 제공할 것인가?
4. AI 채점의 정확도와 신뢰도를 교사가 어떻게 확인할 것인가?
5. 선채점 후 기준화와 상세 루브릭을 어떻게 함께 사용할 것인가?
6. 화면과 버튼을 어떤 순서로 재구성할 것인가?
7. 데이터 손상이나 기능 퇴행 없이 어떤 단계로 구현하고 검증할 것인가?
8. 교내 사용자 시험에서 추가 설문 없이 성능 자료를 수집하고 학생 정보를 어떻게 보호할 것인가?
9. 교사 확정 데이터가 쌓인 뒤 재채점 대상과 채점 기준을 어떻게 교사 판단에 맞게 개선할 것인가?

이 문서는 화면 설계, 데이터 구조, 구현 순서, 테스트 기준, 사용자 매뉴얼 작성의 공통 기준으로 사용한다.

## 2. 개발 원칙과 범위

### 2.1 최우선 목표

1. 학교 수행평가의 보고서·탐구 결과물 채점을 정확하고 편리하게 수행한다.
2. 정기고사의 서술형 문항을 소문항 단위까지 안정적으로 채점한다.
3. AI 채점 결과를 교사가 검토하고 승인한 뒤 확정하는 구조를 제공한다.
4. 여러 차례 독립 채점과 수동 점수 비교를 통해 AI 점수의 신뢰도를 확인할 수 있게 한다.
5. 통합 스캔 PDF를 학생별로 자동 분할해 교사의 사전 작업을 줄인다.
6. 루브릭이 있는 경우에는 정확히 반영하고, 없는 경우에는 자동 생성 또는 선채점 후 기준화를 지원한다.
7. 처음 사용하는 교사도 기술 용어를 몰라도 현재 상태와 다음 할 일을 이해할 수 있게 한다.

### 2.2 현재 우선순위

| 우선순위 | 영역 | 목표 |
|---|---|---|
| P0 | 보고서·수행평가 | 학교 현장에서 실제 채점 완료까지 안정적으로 사용 |
| P0 | 정기고사 서술형 | 학생별·소문항별 정확한 채점과 검토 |
| P0 | 신뢰도 검증 | 회차 비교, 표준편차, 수동 점수 차이, 교사 승인 |
| P0 | 자료 준비 | 통합 스캔 분할, 명렬 관리, 누락·불일치 확인 |
| P1 | 사용 편의 | 단계형 UI, 상태 안내, 도움말과 오류 복구 |
| P1 | 비용·속도 | 모델 선택, 선택 재채점, 예상 비용·시간 안내 |
| P1 | 현장 검증 | 무입력 성능자료 내보내기, 교내 배포용 가명처리 |
| P2 | 대회·탐구 심사 | 순위 조정과 최종 점수 자동 배정 |
| P2 | 확장 공급자 | Claude, ChatGPT 웹 정액제 등 후속 확장 |
| P3 | 적응형 채점 | 축적 데이터 기반 재채점 추천, AI-교사 오차 보정, 자연어 기준 튜닝 |

### 2.3 이번 재구조화에서 하지 않을 일

- 기능 검증이 끝나기 전 EXE를 다시 빌드하지 않는다.
- 기관용 권한 관리, 저장소 암호화, 보존 기간 정책 등 개인정보 보호의 전면 고도화는 별도 단계로 미룬다.
- 단, 교내 사용자 시험에서 외부로 전달되는 개발 피드백 묶음의 가명처리와 원본 자료 기본 제외는 배포 전 필수 범위에 포함한다.
- 기존 데이터를 버리고 새 프로젝트 형식으로 일괄 전환하지 않는다.
- 모든 화면과 백엔드를 한 번에 교체하는 전면 재작성은 하지 않는다.
- 불안정성이 확인된 Gemini 웹 자동화 방식을 기본 실행 경로로 복원하지 않는다.
- 대회 순위 조정 기능을 P0 작업보다 먼저 고도화하지 않는다.
- 교사 확정 데이터가 충분하지 않은 상태에서 기계학습형 자동 보정이나 개인화 모델을 먼저 만들지 않는다.

## 3. 제품 비전

AI 채점기는 “AI에게 파일을 보내 점수를 받는 프로그램”이 아니라, 교사의 평가 업무 전체를 보조하는 로컬 작업공간이어야 한다.

핵심 흐름은 다음과 같다.

> 평가 준비 → 학생·답안 정리 → AI 반복 채점 → 차이와 불확실성 확인 → 교사 검토·승인 → 성적·피드백 내보내기

AI는 최종 결정권자가 아니다. 앱은 AI의 점수뿐 아니라 근거, 회차 간 차이, 수동 점수와의 차이, 검토 필요 여부를 함께 제공해야 한다. 최종 점수는 교사가 승인한 값으로 별도 보존한다.

## 4. 주요 사용자와 사용 상황

### 4.1 핵심 사용자

- 수행평가 보고서를 반 전체 단위로 채점하는 교사
- 정기고사 서술형 답안지를 채점하는 교사
- 과학·수학의 수식, 계산, 그래프, 도식 답안을 검토하는 교사
- 탐구대회나 교내 심사에서 결과물 평가와 순위를 함께 관리하는 교사
- API 설정에 익숙하지 않지만 AI 정액제를 사용하는 교사

### 4.2 대표 사용 상황

#### 상황 A: 공식 루브릭이 있는 보고서 평가

- 교사가 루브릭 문서와 학생별 보고서를 등록한다.
- 앱이 루브릭을 구조화하고 교사가 확인한다.
- 전체 학생을 2회 독립 채점한다.
- 회차 차이가 큰 학생만 추가 채점한다.
- 교사가 결과와 피드백을 검토하고 확정한다.

#### 상황 B: 루브릭이 없는 수행평가

- 교사가 “물리 탐구 보고서, 총 20점, 이론·실험·분석·표현 평가”처럼 간단히 입력한다.
- 앱이 교사용 루브릭 초안을 만든다.
- 교사가 수정·승인한 뒤 채점에 사용한다.

#### 상황 C: 공식 채점기준표가 있는 서술형 시험

- 문제지, 정답지 또는 채점기준표를 등록한다.
- 앱이 대문항·소문항, 배점, 정답, 부분 점수 기준을 추출한다.
- 반 전체 통합 스캔을 학생별로 분할한다.
- 학생 답안만 보고 채점할 수 있도록 각 학생 PDF를 연결한다.
- 소문항별 반복 채점 후 교사가 검토한다.

#### 상황 D: 문제와 정답은 있지만 채점기준표가 없는 시험

- 앱이 모범 답안과 기본 채점기준표를 생성한다.
- 교사가 검토·수정한다.
- 필요하면 일부 답안을 먼저 채점해 실제 답안 유형을 반영한 기준으로 보정한다.

#### 상황 E: 문제 없이 정답과 답안만 있는 시험

- 문제 추정에 과도하게 의존하지 않는다.
- 학생이 실제로 작성한 문항과 정답 자료를 중심으로 자율 선채점한다.
- 문항 번호·배점·필수 기준이 불명확하면 교사 확인 대상으로 표시한다.

#### 상황 F: AI 신뢰도를 먼저 시험하는 교사

- 교사가 일부 학생의 소문항 점수를 수동으로 입력한다.
- 앱이 AI 평균·중앙값, 표준편차, 수동 점수, 차이와 오차 지표를 비교한다.
- 신뢰 가능한 문항과 기준 보정이 필요한 문항을 구분한다.

#### 상황 G: 교내 사용자 시험 결과를 개발자에게 전달

- 교사는 평소처럼 문제·루브릭·학생 자료를 등록하고 AI 채점, 수동 점수 입력, 수정, 승인을 수행한다.
- 별도의 만족도 설문이나 수정 이유 입력을 요구하지 않는다.
- 앱은 이미 발생한 채점 기록과 실행 정보만으로 개발용 성능자료를 만든다.
- 실제 학년·학급·번호·이름은 내보내기 단계에서 원래 정보와 무관한 가상 코드로 대체한다.
- 원본 답안·사진·영상·음원은 표준 피드백 묶음에서 제외한다.
- 교사는 생성된 익명 진단 묶음만 개발자에게 전달한다.

## 6. 현재 상태와 주요 위험

### 6.1 코드 구조 위험

- `app.py`와 `static/app.js`가 매우 커져 기능 간 영향 범위가 넓다.
- HTML, JavaScript, API의 요소 이름이나 기대 데이터가 어긋날 가능성이 있다.
- 여러 AI가 동시에 같은 파일을 수정하면 변경 사항 유실과 기능 퇴행이 발생할 수 있다.
- 프로젝트 데이터 형식이 지속적으로 확장되어 이전 프로젝트 호환성이 중요하다.

### 6.2 테스트 기준선

2026-07-17 최초 기준선 자동 테스트 결과:

- 통과: 36개
- 실패: 5개

확인된 실패 원인은 모두 최근 공급자·모델·데이터 구조 변경 뒤 테스트 기대값이 갱신되지 않은 문제였다.

- 제거된 Gemini 웹 모델을 여전히 기대하는 테스트
- OpenAI 기본 공급자 변경을 반영하지 않은 회차 생성 테스트
- 이전 기본 모델명을 기대하는 테스트
- 이전 웹 공급자 마이그레이션 결과를 기대하는 테스트
- 새 필수 필드인 `q1_has_answer`를 넣지 않은 테스트 데이터

GPT 개발본의 0·1단계 1차 구현 후 결과:

- 통과: 45개
- 실패: 0개
- 기존 프로젝트 호환성: 복사된 8개 프로젝트의 상세 정보와 개요 API를 모두 정상 조회
- Python·JavaScript 문법 검사 및 HTML 중복 ID 검사 통과

위 5개 실패는 변경된 사양을 반영하지 못한 낡은 테스트로 분류하여 현재 지원 사양에 맞게 갱신했다.

현재 HTML과 JavaScript의 정적 ID 연결 검사에서는 누락된 요소가 발견되지 않았다. 다만 이는 실제 클릭·상태 전환·API 연동을 보장하지 않으므로 브라우저 스모크 테스트가 추가로 필요하다.

### 6.3 버전 관리 위험

현재 `.git`이 별도 위치의 Git 저장소를 가리키는 형태이며 연결 상태를 다시 확인해야 한다. 구현 전에 다음 중 하나를 반드시 확보한다.

- 정상 Git 저장소 연결과 기준 커밋
- 또는 프로젝트·데이터를 포함하지 않는 안전한 소스 스냅샷

동시에 두 작업자가 같은 핵심 파일을 수정하지 않도록 구현 단계마다 한 명의 통합 책임자를 정한다.

### 6.4 AI 공급자 위험

- 무료 API는 모델별 일일 요청량과 속도 제한이 작다.
- 유료 API는 비용 관리와 키 설정이 필요하다.
- 웹 정액제 브라우저 자동화는 로그인 상태, UI 변경, 전송 실패, 응답 중단에 취약하다.
- Gemini 웹 자동화에서 응답 도중 중단, 전송 버튼 미작동, 입력창 탐색 실패가 실제로 관찰되었다.

따라서 공급자 계층은 교체 가능하게 유지하되, 차기 안정화의 기본 경로는 검증 가능한 API로 둔다. 웹 정액제 연동은 별도 실험 기능으로 격리하고, 안정성 기준을 통과한 뒤 일반 사용자에게 노출한다.

프로젝트를 연 뒤에는 모든 유형에서 같은 여섯 단계가 보이도록 한다.

1. **개요**
2. **평가기준**
3. **학생·답안**
4. **AI 채점**
5. **검토·확정**
6. **내보내기·분석**

상단에는 프로젝트명, 평가 유형, 저장 상태, 현재 단계, 전체 진행률을 표시한다.

### 8.1 공통 화면 원칙

- 화면마다 가장 중요한 기본 행동 버튼은 하나만 강조한다.
- “다음에 할 일”을 화면 상단에 문장으로 표시한다.
- 완료되지 않은 선행 조건 때문에 버튼을 누를 수 없으면 이유를 바로 표시한다.
- 기술적인 공급자·모델·재시도 간격은 고급 설정에 둔다.
- 교사에게 필요한 회차·차이·검토 상태는 숨기지 않는다.
- 오류는 코드 번호보다 원인, 보존된 작업, 재개 방법을 먼저 안내한다.
- 자동 생성된 기준과 교사가 승인한 기준을 시각적으로 구분한다.

### 9.2 공통 구성

- 대상 학급 또는 평가명

- 평가기준 상태
- 학생·답안 준비 상태
- 채점 회차 진행 상태

- 최근 작업 시각

회차별로 다음 정보를 제공한다.

- 사용한 평가기준 버전
- 공급자와 모델
- 시작·완료 시각
- 완료/실패/재검토 수
- 예상 또는 실제 비용

## 10. 2단계: 평가기준 화면

### 10.1 보고서·수행평가의 기준 준비 경로

교사는 다음 중 하나를 선택한다.

1. 루브릭 문서에서 가져오기
2. 간단한 설명으로 자동 생성
3. 직접 작성
4. 이전 프로젝트의 루브릭 복사

루브릭 자동 생성 입력은 짧고 자연스러운 형태를 기본으로 한다.

예:

> 단진동 탐구 보고서, 총 20점. 이론 이해, 실험 설계, 자료 분석, 결론과 표현을 평가.

생성 후에는 다음을 편집한다.

- 평가 항목명
- 배점
- 수준별 기준
- 필수 확인 요소
- 감점·예외 규칙
- 피드백 관점

### 10.2 정기고사 서술형의 기준 준비 경로

교사는 자료 상황에 따라 다음 경로를 선택한다.

1. 공식 채점기준표 사용
2. 문제와 정답에서 모범 답안·기준 생성
3. 문제만으로 모범 답안·기준 생성
4. 정답 자료 중심으로 기본 기준 생성
5. 교사가 텍스트로 직접 입력
6. 먼저 채점한 뒤 실제 답안으로 기준화

앱은 문항 수와 번호를 먼저 확인하고, 누락 또는 중복을 교사에게 보여준다.

### 10.3 교사용 상세 기준과 AI용 압축 기준 분리

상세한 구조를 앱에 저장하는 것과 긴 기준을 AI에 모두 전달하는 것은 분리한다.

#### 교사용 상세 기준

- 공식 문구
- 세부 채점 요소
- 부분 점수
- 대안 답안
- 흔한 오류
- 예외 규칙
- 실제 답안 사례
- 교사 메모
- 변경 이력

#### AI용 압축 기준

- 정답 또는 기대 핵심
- 3~5개의 핵심 판정 요소
- 반드시 적용할 부분 점수 규칙
- 치명적 오류와 필수 예외
- 출력 형식

화면에서는 두 내용을 나란히 확인하되, 기본 편집 대상은 교사용 상세 기준으로 한다. “AI용 기준 미리보기”에서 실제 전송 내용을 확인할 수 있게 한다.

### 10.4 세 가지 채점 모드

문항 또는 평가 항목별로 다음 모드를 지정할 수 있다.

#### 자율 선채점

- 학생 답안과 총 배점을 중심으로 AI가 평가
- 정답은 제공하되 문제나 세부 기준은 선택
- 초기 답안 유형 파악과 기준 발견에 적합

#### 핵심 기준 채점

- 정답과 3~5개의 핵심 요소만 제공
- 지나치게 긴 루브릭으로 인한 성능 저하를 줄임
- 일반적인 본채점 기본값으로 권장

#### 공식 기준 엄격 적용

- 공식 채점기준표와 부분 점수 규칙을 그대로 적용
- 정기고사나 이의 제기 가능성이 큰 평가에 적합

한 프로젝트 안에서도 문항마다 다른 모드를 사용할 수 있다.

### 10.5 평가기준 버전

평가기준은 덮어쓰지 않고 버전으로 보존한다.

예:

- v1: 공식 자료 추출
- v2: 교사 수정
- v3: 1차 선채점 후 대안 답안 추가
- v4: AI용 기준 압축

각 채점 회차는 사용한 기준 버전을 기록한다. 기준을 바꾼 뒤에는 이전 회차와 새 회차가 혼동되지 않도록 경고한다.

## 11. 선채점 후 기준화

### 11.1 목적

처음부터 지나치게 상세한 기준을 AI에 강제하지 않고, 실제 학생 답안에서 나타난 풀이 방식과 오류 유형을 반영해 기준을 다듬는다.

### 11.2 권장 절차

1. 문제 구조, 배점, 정답 또는 참고 답안을 준비한다.
2. 자율 선채점 또는 핵심 기준 채점으로 1회 진행한다.
3. 가능하면 독립된 2회차를 추가한다.
4. 앱이 답안 유형, 대안 풀이, 흔한 오류, 판독 불가 사례를 묶는다.
5. AI가 교사용 상세 루브릭 보정안을 생성한다.
6. 교사가 새 기준을 승인·수정한다.
7. 앱이 새 기준을 AI용 압축 기준으로 변환한다.
8. 표본 학생 또는 차이가 컸던 문항을 재채점한다.
9. 표준편차와 수동 점수 차이가 개선됐는지 비교한다.
10. 개선이 확인된 경우에만 전체 또는 필요한 대상에 적용한다.

### 11.3 기준화 결과에 포함할 내용

- 실제로 관찰된 정답 표현
- 동일한 의미의 대안 답안
- 풀이 과정은 다르지만 타당한 방법
- 흔한 부분 오류
- 최종 답은 맞지만 논리가 부족한 사례
- 식은 맞지만 단위·부호가 틀린 사례
- 그래프 축·범례·핵심점 누락 사례
- 답안 판독이 불가능한 사례
- 교사 확인이 필요한 경계 사례

### 11.4 안전장치

- AI가 제안한 기준은 교사 승인 전에는 본채점 기준이 아니다.
- 낮은 점수 학생의 오류를 일반 규칙으로 과도하게 확대하지 않는다.
- 소수의 특이 답안을 정답 범위로 자동 편입하지 않는다.
- 기준 변경 전후 결과를 별도 보존한다.
- 기준화 후 무조건 전원 재채점하지 않고 표본 검증을 먼저 한다.

## 12. 3단계: 학생·답안 화면

### 12.1 명렬 등록

지원 경로:

- 표에서 직접 입력
- 이름 목록 붙여넣기
- Excel/CSV 가져오기
- 이전 프로젝트 명렬 재사용
- 파일명에서 이름 추정

학생 정보의 최소 필드는 이름이며, 필요할 때만 번호·학번·반을 사용한다.

### 12.2 답안 연결 상태

학생별로 다음 상태를 표시한다.

- 준비 완료
- 파일 없음
- 이름 불일치
- 중복 파일
- 여러 파일 연결
- 분할 확인 필요
- 제외

교사는 드래그 또는 선택으로 잘못 연결된 파일을 다른 학생에게 옮길 수 있어야 한다.

### 12.3 보고서·수행평가

- 개인 또는 모둠 모드
- 참가자당 한 개 또는 여러 개의 파일
- 폴더 일괄 등록
- HWP, DOC, PDF 등 변환 상태
- 제출 누락자 확인
- 참가자 이름 추정과 수동 교정

### 12.4 정기고사 통합 스캔

통합 스캔과 학생 명렬은 이 단계에 둔다. 교사의 권장 순서는 다음과 같다.

1. 명렬 등록
2. 통합 스캔 PDF 등록
3. 분할 방식 선택
4. 학생별 페이지 미리보기
5. 분할 실행
6. 학생별 PDF 연결 결과 확인
7. 누락·중복·페이지 오류 수정

지원 분할 방식:

- 학생당 고정 페이지 수
- 학생별 페이지 범위 직접 지정
- 빈 페이지 또는 구분지 기준 보조 분할
- 이미 분할된 학생 PDF 사용

분할된 PDF의 저장 위치를 화면에 명확히 표시하고 “폴더 열기”를 제공한다.

### 12.5 답안지 중심 채점

문제 인식이 불완전하더라도 분할된 학생 답안에 실제로 존재하는 문항을 우선 채점할 수 있어야 한다.

- 학생 PDF에서 발견된 문항 번호 표시
- 기준표 문항과 매칭
- 발견되지 않은 문항은 자동으로 “미제출”로 단정하지 않고 확인 대상으로 표시
- 답안 존재 여부 검증
- 문제 자료가 불완전할 때는 정답과 배점 중심의 채점 모드 허용

## 13. 정기고사 문항 구조

### 13.1 대문항과 소문항

소문항은 실제 점수가 부여되는 기본 단위다.

예:

- 1번 총 10점
  - 1-(1) 3점
  - 1-(2) 4점
  - 1-(3) 3점

대문항 점수는 소문항 점수의 합으로 자동 계산한다. 소문항이 없는 문항은 화면에서 한 줄의 평면 문항처럼 표시한다.

### 13.2 답안 유형

각 소문항에는 하나의 주된 채점 유형을 지정한다.

- 단답·수치
- 문장 서술
- 수식·계산
- 그래프·도식

“복합형”을 별도 유형으로 만들지 않는다. 대신 필수 요소를 복수 선택한다.

### 13.3 필수 요소

- 설명
- 근거
- 유도 과정
- 최종 답
- 단위
- 부호
- 수식
- 그래프
- 도식
- 축
- 범례
- 핵심 좌표·값

예를 들어 주된 유형이 “그래프·도식”이면서 설명, 축, 범례, 핵심값을 필수 요소로 설정할 수 있다.

### 13.4 채점 요소와 점수 열

소문항 내부의 채점 요소는 AI가 부분 점수를 설명하기 위한 구조다. 교사의 수동 점수 입력 열을 세부 요소마다 강제하지 않는다.

- AI: 세부 요소별 충족 여부와 근거를 기록 가능
- 교사: 소문항 최종 수동 점수를 입력
- 대문항과 총점: 자동 합산

이 구조는 수동 입력 부담을 줄이면서 AI의 채점 근거를 보존한다.

## 14. 4단계: AI 채점 화면

### 14.1 채점 시작 전 요약

- 채점 대상 수

- 평가기준 버전

### 14.2 회차 실행 전략

1. 전원을 서로 독립된 2회차로 채점

3. 일치하는 결과는 중앙값 또는 합의값으로 집계
4. 교사가 검토 후 최종 확정

프로젝트 상황에 따라 1회, 2회, 3회 이상을 선택할 수 있지만 각 선택의 신뢰도와 비용 차이를 안내한다.

- 현재 회차

- 현재 학생
- 현재 문항 또는 평가 항목
- 성공 수
- 검토 필요 수
- 실패 수
- 재시도 대기 수
- 경과 시간
- 중지 버튼

상태 용어:

- 대기
- 채점 중
- 완료
- 검토 필요
- 실패
- 재시도 중
- 중지됨

### 14.4 실패와 재개

- 성공한 결과를 보존한 채 실패 대상만 재시도
- 중지 후 다음 대상부터 재개
- 공급자·모델을 바꿔 새 회차로 실행
- 모델 변경이 어느 회차부터 적용됐는지 기록
- 쿼터 부족, 인증 오류, 응답 형식 오류, 파일 판독 오류를 구분

## 15. 점수 계층과 데이터 보존

점수는 덮어쓰지 않고 역할별로 분리한다.

### 15.1 공통 점수 계층

1. **회차별 AI 점수**  
   각 회차에서 AI가 제안한 원본 점수

2. **AI 종합 점수**  
   평균, 중앙값 또는 합의 규칙으로 계산한 제안 점수

3. **교사 수동 점수**  
   AI 신뢰도를 비교하기 위해 교사가 독립적으로 입력한 기준 점수

4. **교사 확정 점수**  
   검토와 수정 후 성적에 사용할 최종 점수

### 15.2 대회·탐구 심사의 추가 계층

5. **평가 점수**  
   루브릭에 따라 산출된 원래 점수

6. **순위 조정 후 최종 점수**  
   교사가 순위를 조정하고 허용된 점수표에 맞춰 다시 배정한 점수

평가 점수와 최종 점수는 반드시 모두 보존한다.

### 15.3 변경 이력

다음 변경은 누가, 언제, 무엇을, 왜 변경했는지 기록한다.

- 루브릭 변경
- 수동 점수 입력
- AI 점수 교정
- 교사 확정
- 확정 취소
- 순위 이동
- 최종 점수 수동 변경

## 16. 신뢰도와 수동 점수 비교

### 16.1 학생·소문항별 표시값

- 회차별 AI 점수
- AI 평균
- AI 중앙값
- AI 표준편차
- AI 최솟값·최댓값
- 교사 수동 점수
- AI 대표 점수와 수동 점수의 부호 있는 차이
- 절대 차이
- 검토 필요 여부
- 교사 확정 점수

보고서 평가도 가능한 범위에서 평가 항목별 같은 비교를 제공한다.

### 16.2 전체 분석 지표

- 평균 표준편차
- 평균 절대 오차(MAE)
- 평균 편향: AI 점수 - 수동 점수
- ±1점 이내 비율
- ±2점 이내 비율
- 회차 불일치 학생 수
- 수동 점수 차이 임계값 초과 수
- 문항별 오차 순위
- 모델·기준 버전별 비교

### 16.3 해석 안내

| 회차 간 표준편차 | 수동 점수와 차이 | 해석 | 권장 조치 |
|---|---|---|---|
| 낮음 | 작음 | 안정적이고 교사 판단과 유사 | 일반 검토 후 승인 |
| 낮음 | 큼 | 일관되지만 체계적 편향 가능 | 기준·프롬프트 점검 |
| 높음 | 작음 | 평균은 맞지만 채점이 불안정 | 추가 회차 또는 기준 압축 |
| 높음 | 큼 | 신뢰하기 어려움 | 수동 검토와 기준 재설계 |

### 16.4 자동 검토 표시 조건

- 회차 표준편차가 기준 이상
- 회차별 점수 범위가 기준 이상
- 수동 점수와 차이가 기준 이상
- 답안 존재 여부 판정 불일치
- 판독 불가
- 대안 답안 가능성
- 공식 기준에 없는 풀이
- 배점 범위 오류
- 응답 형식 복구가 발생한 결과

검토 임계값은 프로젝트 수준에서 설정하되 기본값을 제공한다.

### 16.5 중장기 발전: 데이터 축적 기반 적응형 채점

이 기능은 현재 핵심 채점 흐름을 안정화한 뒤, 여러 교사가 실제 자료를 채점하고 수동 점수·수정·승인 기록이 충분히 쌓였을 때 시작한다. 초기에는 설명 가능한 규칙 기반으로 구현하고, 데이터가 충분해진 뒤에만 통계적 보정이나 학습 기반 추천을 검토한다.

#### 현장 비교 데이터가 보여준 필요

`output/채점모드_비교_2026과학올림피아드.xlsx`의 13명×7문항 비교에서는 자율·핵심·엄격 세 모드의 전체 순위는 대체로 비슷했지만 다음 현상이 나타났다.

- 91개 학생-문항 중 모드 간 5점 이상 차이 10건
- 답안 존재 여부 판정 불일치 3건
- 학생 총점의 모드 간 범위가 평균 8점
- `검토 필요`가 모드별 74~78%로 지나치게 많이 표시
- 엄격 모드가 일부 학생의 총점과 순위를 크게 낮춤

이 실험은 같은 모드의 반복 재현성 시험이 아니라 서로 다른 채점 정책의 민감도 비교다. 따라서 서로 다른 모드의 점수를 평균하지 않고, 먼저 교사 수동 점수로 기본 채점 정책을 정한 뒤 같은 정책의 반복 안정성을 별도로 검증한다.

#### 발전 목표

1. 재채점이 실제로 필요한 답안만 정확하게 골라 비용과 시간을 줄인다.
2. AI 점수와 교사 수동 점수 사이의 반복되는 차이를 찾아 문항·기준별 편향을 줄인다.
3. 교사가 자연어로 원하는 채점 원칙을 설명하면 구조화된 기준과 프롬프트로 안전하게 변환한다.
4. 교사별·과목별 판단을 존중하되 근거 없는 자동 개인화와 과적합을 막는다.
5. 모든 추천과 보정은 이유, 적용 범위, 버전, 변경 전후 결과를 교사가 확인할 수 있게 한다.

### 16.6 적응형 재채점 대상 선정

재채점 추천은 하나의 확신도 값에 의존하지 않고 여러 신호를 결합한다.

#### 입력 신호

- 같은 모드 반복 회차의 점수 범위와 표준편차
- 서로 다른 모델 또는 공급자 간 점수 차이
- 답안 존재 여부 판정 불일치
- OCR·수식·그래프 판독 불확실성
- AI 확신도와 근거의 구체성
- 문항별 과거 교사 수정률과 평균 오차
- 교사 수동 점수와의 예상 차이
- 총점·등급·수상권 경계까지의 거리
- 대안 풀이, 공식 기준 밖 풀이, 응답 형식 복구 여부
- 동일 답안군에서 벗어나는 이상 점수

#### 출력 단계

- `일반 검토`: 한 회차 결과를 교사가 통상 확인
- `선택 재채점`: 추가 AI 회차가 비용 대비 유효할 가능성이 높음
- `즉시 교사 검토`: 답안 판독·존재 여부·평가 정책 판단이 필요
- `자동 확정 후보`: 충분한 검증 후에만 제공하며 최종 결정은 교사에게 있음

각 추천에는 “2회차 점수 차이 6점”, “답안 존재 판정 불일치”, “이 문항의 과거 교사 수정률 28%”처럼 사람이 이해할 수 있는 이유를 표시한다.

#### 단계적 알고리즘

1. 고정 임계값 규칙으로 시작한다.
2. 실제 교사 확정 데이터로 임계값별 적중률과 불필요 재채점률을 측정한다.
3. 과목·문항 유형별로 다른 임계값을 제안한다.
4. 충분한 데이터가 쌓이면 재채점 필요 확률을 예측하는 통계 모델을 검토한다.
5. 새 알고리즘은 기존 규칙과 병행 비교한 뒤 교사가 승인해야 기본값이 된다.

### 16.7 AI-교사 점수 차이 보정

교사가 확정한 점수를 정답 데이터로 사용해 다음 단위의 오차를 추적한다.

- 프로젝트
- 평가기준 버전
- 과목과 평가 유형
- 대문항·소문항 또는 루브릭 항목
- 답안 유형: 단답, 문장, 수식, 그래프·도식, 복합
- 공급자와 모델

주요 분석 지표:

- 평균 절대 오차(MAE)
- 부호 있는 평균 오차와 과대·과소 채점 비율
- 정확 일치율과 ±1점·±2점 이내 비율
- 문항별 교사 수정률
- 높은 확신인데 틀린 비율
- 확신도 임계값별 자동 처리 가능 비율
- 재채점으로 실제 오류를 발견한 비율
- 교사 검토 시간과 AI 비용 절감률

오차가 반복되면 점수를 기계적으로 더하거나 빼지 않고 다음 순서로 개선한다.

1. 판독 오류와 기준 적용 오류를 분리한다.
2. AI용 압축 기준, 부분 점수 규칙, 대안 답안 사례를 보정한다.
3. 표본 답안에서 변경 전후 결과를 비교한다.
4. 교사가 개선안을 승인한다.
5. 새 기준 버전으로 필요한 대상만 재채점한다.
6. 개선되지 않으면 이전 버전으로 되돌린다.

교사별 보정과 공통 제품 개선 데이터는 분리한다. 한 교사의 개인적 판단을 다른 학교·교사에게 자동 적용하지 않는다.

### 16.8 자연어 채점 기준 튜닝

평가기준 화면에 `교사 방식으로 조정` 입력란을 제공한다.

입력 예:

> 최종 식은 맞지만 단위가 빠지면 1점 감점. 계산 실수는 이후 과정이 일관되면 70%까지 인정. 공식 답과 다른 풀이도 물리적으로 타당하면 인정.

앱은 이 문장을 바로 전체 채점 프롬프트에 덧붙이지 않고 다음 구조로 변환한다.

- 적용 대상 문항·평가 항목
- 핵심 판정 규칙
- 부분 점수와 감점 규칙
- 인정할 대안 답안
- 인정하지 않을 오류
- 판독 불확실 시 처리
- 규칙 간 충돌과 모호한 표현

적용 절차:

1. 교사의 자연어 지시를 구조화된 기준 초안으로 변환한다.
2. 현재 기준과의 차이를 표시한다.
3. AI에게 실제 전달될 압축 프롬프트를 미리 보여준다.
4. 대표 표본 답안에 `시험 적용`한다.
5. 교사 점수와 변경 전후 오차를 비교한다.
6. 교사가 승인한 경우에만 새 기준 버전으로 저장한다.
7. 전체 재채점 대신 영향을 받을 가능성이 큰 답안부터 선택 적용한다.

교사의 자연어 지시는 평가기준 버전에 포함하며, 언제든 이전 버전으로 되돌릴 수 있어야 한다.

### 16.9 데이터 준비와 도입 조건

#### 지금부터 저장할 데이터

- 회차별 원점수, 근거, 확신도, 답안 존재 판정
- 공급자·모델·프롬프트·평가기준 버전
- 교사 수동 점수와 최종 확정 점수
- AI 점수 수정 전후 값과 승인 이벤트
- 재채점 추천 이유와 교사의 실제 조치
- 처리 시간, 비용, 실패·재시도 기록

별도 피드백 설문이나 필수 수정 이유 입력은 요구하지 않는다. 사용 과정에서 자연스럽게 생긴 기록을 우선 사용한다.

#### 도입 순서

1. 데이터 수집과 지표 계산
2. 설명 가능한 고정 규칙 기반 추천
3. 프로젝트·문항별 임계값 보정
4. 교사 자연어 기준의 구조화와 표본 시험
5. 충분한 교사 확정 데이터가 있는 영역부터 통계 기반 추천
6. 여러 학교에 적용하기 전 별도 검증 데이터로 일반화 성능 확인

데이터 양만으로 자동 전환하지 않는다. 답안 유형의 다양성, 교사 점수의 일관성, 다른 학급·시기의 검증 결과를 함께 확인한다.

#### 개인정보 원칙

- 적응형 기능의 공유·분석 데이터는 18.6절의 가명처리 원칙을 따른다.
- 원본 답안은 교사 PC의 로컬 분석에서만 사용하고 표준 피드백 묶음에는 포함하지 않는다.
- 가명 데이터와 실제 학생 대응표를 분리한다.
- 학교·교사별 보정 프로필은 명시적으로 구분하고 다른 사용자에게 섞이지 않게 한다.

## 17. 5단계: 검토·확정 화면

### 17.1 공통 원칙

- 검토가 필요한 결과를 먼저 보여준다.
- AI 원본 결과와 교사 수정 결과를 모두 보존한다.
- 학생별 보기와 평가 단위별 보기를 모두 제공한다.
- 교사 승인 전 결과는 “제안 점수”다.
- 일괄 승인은 검토 필요 항목이 없는 범위에서만 허용한다.

### 17.2 보고서·수행평가 검토

왼쪽:

- 학생 또는 모둠 원본 자료
- 페이지 이동과 확대

오른쪽:

- 평가 항목별 회차 점수
- AI 종합 점수
- 근거
- 피드백
- 종합 의견
- 세특 문장
- 교사 수정
- 확정 버튼

### 17.3 정기고사 검토

두 가지 관점을 제공한다.

#### 소문항별 보기 — 권장 기본값

- 한 소문항에 대한 전체 학생 답안과 점수를 연속 비교
- 같은 기준이 일관되게 적용됐는지 확인
- 수동 점수 차이와 표준편차로 정렬
- 경계 답안끼리 비교

#### 학생별 보기

- 한 학생의 모든 소문항
- 대문항 합계와 총점
- 회차별 결과
- 답안 존재 여부
- 근거 문구
- 교사 수정과 확정

### 17.4 승인 상태

- 미검토
- 검토 필요
- 검토 완료
- 확정
- 확정 후 수정

확정 후 수정 시에는 사유를 입력하고 변경 이력을 남긴다.

## 18. 6단계: 내보내기·분석 화면

### 18.1 공통 내보내기

- 학생별 최종 점수
- 항목별 또는 소문항별 점수
- AI 점수와 교사 확정 점수
- 피드백과 종합 의견
- 검토 상태
- 회차별 결과
- 수동 점수 비교
- 분석 지표

### 18.2 보고서 평가

- 루브릭 항목별 점수표
- 종합 피드백
- 세특 문장
- 제출 점수
- 대회 모드의 평가 점수와 최종 점수

### 18.3 정기고사

- 소문항별 점수
- 대문항 합계
- 총점
- 문항별 평균과 표준편차
- 수동 점수 차이
- 검토 대상 목록

### 18.4 내보내기 원칙

- 무엇을 내보낼지 교사가 선택
- 개인정보가 포함됨을 명확히 안내
- 확정 전 점수와 확정 점수를 구분
- 파일명에 프로젝트명, 버전, 날짜 포함
- 같은 파일을 덮어쓰기 전에 확인

### 18.5 개발용 성능자료 내보내기

교내 사용자 시험에서 받는 피드백은 별도 설문이 아니라 실제 사용 중 자동으로 쌓인 기록을 이용한다.

#### 무입력 원칙

- 사용자는 채점에 필요한 정보 외에 개발 피드백을 위한 내용을 추가로 작성하지 않는다.
- “정확했나요?”, “수정 이유를 작성하세요” 같은 필수 질문을 띄우지 않는다.
- 수동 점수 입력, 점수 수정, 재채점, 검토와 승인을 암묵적 피드백으로 해석한다.
- 기존 교사 메모가 있으면 포함하되 메모가 없어도 정상적으로 생성한다.
- 수동 점수가 없으면 정확도는 `측정 불가`로 표시하고 회차 일관성, 실패율, 처리 시간 등 계산 가능한 항목만 내보낸다.
- 총점 수동 점수만 있으면 총점 비교만, 항목·소문항 점수가 있으면 세부 정확도까지 계산한다.

#### 자동 수집 항목

- 앱 버전과 데이터 스키마 버전
- 프로젝트 유형과 평가 단위
- 공급자, 모델, 평가기준 버전
- 회차별 AI 점수, 평균·중앙값·표준편차
- AI 확신도와 검토 필요 판정
- 교사 수동 점수와 최종 확정 점수
- AI 점수와 교사 점수 차이
- 점수 수정 전후 값, 수정 횟수, 승인 상태
- 재채점, 실패, 재시도, 중단·재개 기록
- 처리 시간, 요청 수, 확인 가능한 비용
- 입력 자료의 형식, 크기, 페이지 수 또는 재생 시간

#### 표준 산출물

- `성능요약.xlsx`: 교사가 읽을 수 있는 점수·오차·수정률·실패율 요약
- `채점데이터.jsonl`: 학생·항목·소문항·회차 단위의 분석용 데이터
- `실행진단.json`: 앱·모델·기준 버전, 처리 시간, 오류와 재시도 기록
- `자료목록.csv`: 원본을 포함하지 않은 파일 형식·크기·페이지·재생 시간 목록
- `안내.txt`: 포함 항목, 제외 항목, 가명처리 방식, 정확도 계산 가능 범위

UI에서는 일반 성적표와 혼동되지 않도록 다음 두 작업을 구분한다.

- `교사용 결과표 내보내기`
- `개발용 성능자료 내보내기`

### 18.6 개발 피드백 묶음의 가명처리

#### 식별정보 대체

- 실제 학년, 학급, 번호, 이름은 교사 PC의 원본 프로젝트에만 유지한다.
- 피드백 묶음에서는 실제 정보와 무관한 가상 학년, 가상 학급, 무작위 학생 코드를 사용한다.
- 이름을 `김*수`처럼 일부 마스킹하지 않고 `학생 017` 또는 무작위 가명으로 완전히 대체한다.
- 실제 번호에 일정 값을 더하는 방식이 아니라 프로젝트 또는 내보내기 단위로 새 번호를 무작위 배정한다.
- 같은 묶음 안에서는 회차 간 비교를 위해 같은 가상 ID를 유지한다.
- 서로 다른 프로젝트에서 같은 학생을 연결할 필요가 없으면 프로젝트마다 새 가상 ID를 만든다.
- 실제 정보와 가상 ID의 대응표는 로컬에 별도 보관하고 피드백 묶음에 절대 포함하지 않는다.

예:

| 로컬 프로젝트 표시 | 개발 피드백 묶음 |
|---|---|
| 2학년 1반 12번 김민수 | 가상학년 A · 가상반 B · 학생 017 |
| 실제 학생 ID | `S-017` |
| `12_김민수_답안.pdf` | `S-017_answer.pdf` |

#### 텍스트·경로·메타데이터 정리

- AI 답안 요약, 채점 근거, 교사 메모, 오류 로그에 포함된 실제 이름과 번호도 치환한다.
- 전화번호, 이메일, 주소, 계정 ID와 개인을 특정하기 쉬운 고유 활동명도 탐지·제거 대상으로 둔다.
- 학교명, 교사명, 컴퓨터 사용자명, 절대 경로, 원래 파일명을 제거한다.
- API 키, 인증 정보, 브라우저 프로필 정보는 어떤 경우에도 포함하지 않는다.
- 파일 메타데이터와 사진의 EXIF 정보는 포함하지 않는다.
- 프로젝트명이나 평가명에 학교·학급 정보가 있으면 일반화된 이름으로 바꾼다.

#### 원본 자료 처리

- 표준 개발 피드백 묶음에는 원본 PDF, 답안 이미지, 미술 작품 사진, 영상, 음원을 포함하지 않는다.
- 원본을 제외해도 AI-교사 점수 차이, 회차 일관성, 수정률, 오류율, 속도와 비용은 분석할 수 있어야 한다.
- OCR·판독 실패처럼 원본 확인이 꼭 필요한 사례는 표준 묶음과 분리된 `문제 사례 익명 사본` 기능으로 후속 제공한다.
- 문제 사례 사본은 이름 영역, 서명, 얼굴, 명찰, 학교 로고, 음성 속 이름, 파일 메타데이터 등 비정형 식별정보를 처리하고 교사가 최종 확인한 뒤에만 내보낸다.
- 자동 마스킹은 완전성을 보장할 수 없으므로 원본 포함은 기본값이 될 수 없다.

#### 법·정책상 취급 원칙

- 가상 코드로 대체한 자료도 재식별 가능성이 있으면 완전한 익명정보가 아니라 가명정보로 취급한다.
- 데이터 최소화, 대응표 분리 보관, 접근 제한, 목적 달성 후 파기를 기본 원칙으로 삼는다.
- 학교 내부 배포·제공 권한과 개인정보 처리 근거는 실제 배포 전에 학교 정책과 최신 지침으로 확인한다.
- 참고 기준: [개인정보 보호법 제2조](https://www.law.go.kr/LSW/lsSideInfoP.do?docCls=jo&joBrNo=00&joNo=0002&lsiSeq=270351&urlMode=lsScJoRltInfoR), 개인정보보호위원회 [「가명정보 처리 가이드라인(2026.3.)」](https://m.pipc.go.kr/np/cop/bbs/selectBoardArticle.do?bbsId=BS217&mCode=D010030000&nttId=11931)

## 19. 프로젝트 생성 마법사

새 프로젝트 생성은 다음 순서로 구성한다.

### 19.1 1단계: 평가 유형

- 보고서·수행평가
- 대회·탐구 심사
- 정기고사 서술형

각 유형에 한 문장의 설명과 대표 예시를 표시한다.

### 19.2 2단계: 기본 정보

- 프로젝트명
- 학년·반 또는 대상
- 평가명
- 총점
- 개인/모둠
- 예상 인원

### 19.3 3단계: 자료 상황

보고서:

- 루브릭 있음
- 간단한 설명으로 생성
- 나중에 준비

서술형:

- 공식 채점기준표 있음
- 문제와 정답 있음
- 문제만 있음
- 정답 자료 중심
- 먼저 답안을 보고 기준화

### 19.4 4단계: AI 사용 방식

- 기본 권장 설정 사용
- 고급 설정 직접 선택

기본 권장 설정은 공급자와 모델을 선택할 수 있되, 설정의 세부 기술 용어는 숨긴다. API 키가 없으면 설정 화면으로 안내한다.

### 19.5 생성 완료

생성 후 개요 화면으로 이동하고 “다음 할 일”을 표시한다. 생성 직후 모든 탭을 한꺼번에 설명하지 않는다.

### 20.1 일반 설정

- 자동 저장

### 20.2 AI 연결

- API 키

- 기본 회차 수
- 추가 회차 조건

- 저장할 때 스키마 버전을 기록한다.
- 마이그레이션 전 프로젝트 백업을 만든다.
- 마이그레이션은 여러 번 실행해도 같은 결과가 되도록 한다.
- 기존 프로젝트를 열고 저장하지 않아도 조회 가능해야 한다.

### 22.9 개발 피드백 내보내기 모델

표준 피드백 묶음은 원본 프로젝트의 복사본이 아니라 분석에 필요한 최소 필드로 새로 생성한다.

- 내보내기 ID와 생성 시각
- 내보내기 스키마 버전
- 익명 데이터셋 ID
- 가상 학년·학급·학생 ID
- 프로젝트·평가 유형
- 공급자·모델·기준 버전
- 실행·회차 ID
- AI 점수·근거·확신도·검토 필요
- 수동 점수·확정 점수·점수 차이
- 수정·승인·재시도 이벤트
- 처리 시간·요청 수·오류 분류·비용
- 원본을 제외한 자료 형식 메타데이터

로컬에만 남기는 정보:

- 실제 학생 ID, 학년, 학급, 번호, 이름
- 실제 정보와 가상 ID의 대응표
- 원본 파일과 원래 파일명·경로
- API 키와 인증 정보

내보내기 생성기는 허용 필드 목록 방식으로 구현한다. 원본 프로젝트 객체를 그대로 직렬화한 뒤 일부 필드만 지우는 방식은 사용하지 않는다.

### 22.10 적응형 채점과 보정 프로필

장기 고도화를 위해 다음 데이터를 별도 버전으로 관리한다.

- 보정 프로필 ID
- 적용 범위: 프로젝트 / 교사 / 과목 / 문항 유형
- 기반 평가기준과 프롬프트 버전

- 기준 데이터의 기간과 사례 수
- AI-교사 오차 지표

- 활성·비활성·이전 버전

보정 프로필은 기존 점수를 직접 덮어쓰지 않는다. 새 회차나 새 기준 버전에만 명시적으로 적용하고, 적용 전후 결과를 모두 보존한다.

## 24. 구현 단계

각 단계는 “기능 구현 → 자동 검사 → 교사 수동 검증 → 다음 단계” 순서로 진행한다.

### 0단계: 기준선 고정

#### 작업

- 다른 AI의 현재 작업 완료 확인
- 소스 기준 버전 또는 스냅샷 확보
- 현재 프로젝트 데이터 백업
- 자동 테스트 5개 실패 분류와 갱신
- 주요 기존 기능 스모크 테스트
- 기존 프로젝트 2~3개의 호환성 확인

#### 완료 조건

- 기준 소스가 명확하다.
- 알려진 실패 목록이 문서화됐다.
- 기존 보고서 프로젝트와 서술형 프로젝트를 열 수 있다.
- 새 프로젝트 생성과 기존 결과 조회가 가능하다.

### 1단계: 공통 앱 셸과 프로젝트 생성

#### 작업

- 세 가지 시작 유형 구현
- 여섯 단계 공통 탐색 구조
- 개요 화면
- 준비 상태와 다음 행동 안내
- 유형별 기능 표시 규칙

#### 완료 조건

- 처음 사용하는 교사가 설명 없이 세 유형을 구분한다.
- 프로젝트 생성 후 다음 단계가 명확하다.
- 기존 프로젝트가 적절한 유형으로 열린다.
- 프로젝트 유형 버튼과 탭 이동이 자동 테스트를 통과한다.

#### GPT 개발본 1차 구현 기록 — 2026-07-17

- 프로젝트 스키마를 v4로 올리고 보이는 업무 유형 `workflow_type`과 내부 채점 엔진 `project_type`을 분리했다.
- 보고서·수행평가와 대회·탐구 심사는 기존 보고서 엔진을 공유하고, 정기고사 서술형은 시험 엔진을 사용한다.
- 기존 프로젝트는 저장 파일을 즉시 다시 쓰지 않고 열 때 유형을 안전하게 추론한다.
- 네 단계 프로젝트 생성 마법사와 세 가지 유형 선택 카드를 추가했다.
- 프로젝트를 연 뒤 개요, 평가기준, 학생·답안, AI 채점, 검토·확정, 내보내기·분석의 여섯 단계가 항상 보이도록 앱 셸을 구성했다.
- 개요 API와 화면에서 평가기준·답안·회차·검토 상태를 계산하고 다음 권장 작업을 안내한다.
- 채점 화면에 현재 회차와 완료·성공·실패 수를 계속 노출한다.
- 기존 결과·분석·순위 조정 기능은 삭제하지 않고 새 단계에 연결했다.
- 자동 테스트 45개와 기존 프로젝트 8개 호환성 검사는 통과했다.
- 교사 브라우저 수동 검증 후 1단계 완료 여부를 확정한다.

### 2단계: 학생·답안 통합

#### 작업

- 명렬과 통합 스캔을 학생·답안 단계로 이동
- 보고서 제출물과 시험 답안의 공통 연결 상태
- 통합 PDF 분할 흐름 정리
- 분할 저장 위치와 폴더 열기
- 누락·중복·불일치 교정 UI

#### 완료 조건

- 명렬 등록부터 학생 PDF 생성까지 한 화면 흐름으로 완료한다.
- 이미 분할된 PDF도 사용할 수 있다.
- 잘못 연결된 학생 파일을 교정할 수 있다.
- 기존 분할 결과를 새 구조에서 다시 사용한다.

### 3단계: 평가기준 통합

#### 작업

- 보고서 루브릭 경로 통합
- 서술형 공식 기준·자동 생성·텍스트 입력 통합
- 대문항·소문항 구조
- 답안 유형과 필수 요소
- 교사용 상세 기준과 AI용 압축 기준
- 기준 버전과 승인

#### 완료 조건

- 공식 기준이 있는 경우와 없는 경우를 모두 처리한다.
- 소문항 배점 합이 대문항 배점과 맞지 않으면 경고한다.
- 상세 기준과 실제 AI 전송 기준을 구분해 확인할 수 있다.
- 기존 루브릭 데이터가 손실 없이 열린다.

### 4단계: 채점 회차와 진행 화면

#### 작업

- 회차를 개요와 채점 화면에 전면 표시
- 2회 독립 채점과 차이 대상 3회차
- 공급자·모델·기준 버전 기록
- 성공 결과 보존형 재개
- 실패 유형별 안내
- 예상 요청 수·비용·시간

#### 완료 조건

- 현재 몇 회차의 몇 번째 학생인지 즉시 알 수 있다.
- 모델 변경이 새 회차에 실제 적용됐음을 확인할 수 있다.
- 중단 후 성공 결과를 잃지 않고 재개한다.
- 실패 학생만 재시도할 수 있다.

### 5단계: 검토·확정과 수동 점수 비교

#### 작업

- 회차별 점수와 AI 종합 점수
- 수동 점수 입력·가져오기
- 표준편차와 차이 지표
- 보고서 자료 병렬 검토
- 서술형 소문항별·학생별 검토
- 교사 승인과 변경 이력

#### 완료 조건

- AI 점수, 표준편차, 수동 점수, 차이를 한 화면에서 본다.
- 교사 수동 점수가 자동으로 확정 점수를 덮어쓰지 않는다.
- 검토 필요 결과를 우선 처리한다.
- 확정 후 수정 이력이 보존된다.

### 6단계: 선채점 후 기준화

#### 작업

- 선채점 결과 묶음 분석
- 대안 답안·오류 유형 제안
- 상세 루브릭 보정안
- AI용 압축 기준 생성
- 표본 재채점
- 기준 변경 전후 신뢰도 비교

#### 완료 조건

- 교사 승인 전 기준이 본채점에 적용되지 않는다.
- 기준 변경 전후 결과가 모두 남는다.
- 표본 재채점으로 개선 여부를 확인한다.
- 전체 재채점은 교사가 명시적으로 선택한다.

### 7단계: 분석·내보내기

#### 작업

- 공통 점수·신뢰도 분석
- 문항별·평가 항목별 분석
- 보고서 피드백·세특 내보내기
- 서술형 소문항 점수표
- 확정 점수 중심 Excel
- 추가 설문 없는 개발용 성능자료 묶음
- 가상 학년·학급·학생 ID 생성과 로컬 대응표 분리
- 텍스트·파일명·경로·메타데이터의 식별정보 제거
- 원본 답안·이미지·영상·음원의 표준 묶음 제외
- 수동 점수가 없는 경우의 부분 분석과 `측정 불가` 처리

#### 완료 조건

- 현재 기능의 Excel 출력이 빠짐없이 새 구조로 이동한다.
- 확정 전·후 점수가 명확히 구분된다.
- AI-수동 비교 지표가 화면과 파일에서 일치한다.
- 사용자가 채점 외 정보를 추가 입력하지 않아도 성능자료 묶음이 생성된다.
- 실제 이름·학년·학급·번호·학교명·절대 경로·API 키가 묶음에 포함되지 않는다.
- 같은 묶음 안에서는 학생 가상 ID가 일관되고 로컬 대응표는 포함되지 않는다.
- 원본 자료가 기본적으로 제외되며 분석 가능 범위가 안내 파일에 기록된다.

### 8단계: 대회 순위 조정

#### 작업

- 평가 점수 보존
- 교사 드래그 순위 조정
- 허용 점수표에 따른 자동 점수 배정
- 수동 예외 처리
- 조정 이력

#### 완료 조건

- 원래 평가 점수가 변하지 않는다.
- 순위와 최종 점수의 변경 이력을 확인한다.
- 동점과 예외 점수를 처리한다.

### 9단계: 도움말과 배포 준비

#### 작업

- 첫 실행 안내
- 화면별 도움말
- 예제 프로젝트
- 오류 해결 안내
- 교사용 빠른 시작 설명서
- 전체 회귀 테스트
- 사용자 실제 업무 검증
- 마지막 단계에서만 EXE 빌드

#### 완료 조건

- 처음 사용하는 교사가 매뉴얼만 보고 대표 업무를 완료한다.
- 기존 프로젝트와 새 프로젝트의 회귀 테스트를 통과한다.
- 사용자 수동 검증에서 치명적 오류가 없다.
- 그 이후에만 배포 파일을 생성한다.

### 10단계: 데이터 축적 기반 적응형 채점 — 배포 후 장기 고도화

이 단계는 vNext 첫 배포의 완료 조건이 아니다. 교내 사용자 시험과 실제 채점에서 교사 확정 데이터가 축적된 뒤 별도 개발 주기로 시작한다.

#### 진입 조건

- 공급자·모델·기준 버전이 모든 회차에 기록된다.
- 문항별 또는 평가 항목별 교사 확정 점수가 축적된다.
- 교사 수정과 승인 이벤트를 재현할 수 있다.
- 가명 성능자료 묶음이 안정적으로 생성된다.
- 같은 채점 정책의 반복 안정성 기준선이 확보된다.

#### 작업

- 재채점 추천 규칙과 우선순위 점수
- 검토 필요 사유 세분화
- AI-교사 오차 대시보드
- 문항·과목·교사 범위별 보정 프로필
- 자연어 채점 기준 구조화
- 표본 시험과 변경 전후 비교
- 선택 재채점 비용·효과 분석
- 충분한 데이터 영역의 통계 기반 추천 실험

#### 완료 조건

- 재채점 추천 이유를 교사가 이해할 수 있다.
- 고정 2·3회 전체 채점보다 적은 요청으로 주요 오류를 같은 수준 이상 발견한다.
- 새 기준이 별도 검증 표본에서 AI-교사 오차를 줄인다.
- 높은 확신인데 틀린 비율이 악화되지 않는다.
- 교사 승인 없이 보정 규칙이나 기준이 자동 적용되지 않는다.
- 학교·교사별 프로필이 다른 사용자 데이터와 섞이지 않는다.
- 모든 변경을 버전 비교하고 되돌릴 수 있다.

- 프로젝트 생성과 구형 프로젝트 로드
- 경로 이식성
- 루브릭 구조 변환
- 대문항·소문항 합계
- 회차 집계와 표준편차
- AI-수동 점수 차이
- 추가 회차 대상 선정
- 점수 계층 보존
- 승인 이력
- 공급자·모델 설정 적용

### 25.2 API 통합 테스트

- 자료 등록·삭제
- PDF 분할 미리보기·실행
- 기준 추출·생성·기준화
- 회차 생성·시작·중지·재개
- 결과 저장·수정·승인
- 수동 점수 업로드
- 분석과 Excel 내보내기
- 개발용 성능자료 묶음 생성
- 허용 필드 기반 가명 내보내기

외부 AI 호출은 기본적으로 모의 응답을 사용하고, 별도의 소수 실제 호출 테스트를 둔다.

### 25.3 브라우저 스모크 테스트

최소 시나리오:

1. 세 프로젝트 유형 버튼 클릭
2. 새 프로젝트 생성
3. 여섯 단계 이동
4. 조건에 따른 버튼 활성화
5. 파일 등록
6. 명렬 입력
7. 분할 미리보기
8. 채점 시작·중지
9. 결과 열기·수정·승인
10. Excel 내보내기

HTML 요소 존재 검사뿐 아니라 실제 클릭과 화면 상태를 검증한다.

### 25.4 데이터 회귀 테스트

실제 개인정보를 제거한 대표 샘플을 준비한다.

- 기존 보고서 프로젝트
- 기존 서술형 프로젝트
- 통합 스캔 프로젝트
- 여러 회차가 있는 프로젝트
- 수동 점수가 있는 프로젝트
- 승인 결과가 있는 프로젝트
- 부분 실패 후 재개된 프로젝트

### 25.5 개인정보·가명처리 회귀 테스트

- 실제 이름, 학년, 학급, 번호가 모든 산출물에서 제거되는지 검사
- AI 답안 요약·근거·교사 메모·로그 안의 이름도 치환되는지 검사
- 학교명, 사용자명, 절대 경로, 원래 파일명이 제거되는지 검사
- API 키와 인증 정보가 포함되지 않는지 검사
- 같은 묶음 안에서 가상 학생 ID가 회차별로 일치하는지 검사
- 다른 프로젝트에서는 동일 학생이라도 기본적으로 다른 ID가 생성되는지 검사
- 실제-가상 대응표가 ZIP에 포함되지 않는지 검사
- 원본 PDF·이미지·영상·음원이 표준 묶음에 포함되지 않는지 검사
- 수동 점수가 없어도 추가 입력 요구 없이 부분 분석이 생성되는지 검사
- 생성된 ZIP 내부 전체 텍스트를 대상으로 금지 패턴 검사를 수행

### 25.6 교사 수동 검증

개발자가 자동으로 긴 실제 AI 채점 테스트를 반복하기보다, 기능 단위 구현 후 교사가 브라우저에서 짧은 시나리오를 검증한다.

각 검증 요청에는 다음을 제공한다.

- 실행 방법
- 사용할 테스트 자료
- 눌러야 할 버튼
- 기대 결과
- 실패 시 보내야 할 화면·로그

### 25.7 성능·비용 검증

- 학생 5명 소규모
- 한 반 30명 일반 규모
- 여러 파일 보고서
- 수식·그래프 포함 서술형
- 2회차와 선택 3회차

측정 항목:

- 준비 시간
- 학생당 채점 시간
- 요청 수
- 비용
- 실패율
- 재시도율
- 교사 검토 시간

### 25.8 적응형 채점 검증 — 장기 단계

- 같은 채점 정책을 반복 실행한 재현성 데이터와 서로 다른 정책 비교 데이터를 구분
- 과거 자료로 임계값을 정하고 이후 학급·평가 자료로 별도 검증
- 고정 2회 전체 채점과 `1회 전체 + 선택 재채점`의 오류 발견률·비용·검토 시간 비교
- 재채점 추천 건수 중 실제 교사 수정이 발생한 비율
- 재채점하지 않은 답안에서 놓친 큰 오차 비율
- 자연어 기준 변경 전후의 MAE, 편향, 높은 오확신 비율 비교
- 특정 교사·문항에 맞춘 보정이 다른 학급에서 악화되지 않는지 검사
- 데이터가 적거나 분포가 달라지면 기존 규칙으로 안전하게 되돌아가는지 검사
- 교사 승인 전에는 새 기준·보정 프로필이 실제 점수에 적용되지 않는지 검사

## 31. 구현 전 남은 세부 결정

아래 항목은 구현을 막는 핵심 쟁점은 아니며 각 단계 직전에 확정할 수 있다.

1. 기본 회차 수를 항상 2회로 할지 프로젝트 유형별로 다르게 할지
2. 추가 회차 대상의 기본 표준편차·점수 범위 임계값
3. AI 종합 점수의 기본값을 중앙값으로 할지 합의 규칙으로 할지
4. 수동 점수와 AI 점수 차이의 기본 경고값
5. 보고서 평가 항목의 수동 점수를 항목별로 받을지 총점만도 허용할지
6. 소문항 번호 표기 방식: `1-(1)`, `1-1`, `1a` 등의 내부 정규화
7. 통합 스캔의 자동 분할 보조 기능을 어느 수준까지 제공할지
8. 비용 예상이 불가능한 모델의 표시 방식
9. 기준화에 사용할 표본 선정 방식
10. 교사 확정 후 일괄 수정 권한과 확인 절차
11. 대회 순위 조정의 동점 처리 규칙
12. 문제 사례 익명 사본에 원본 자료를 포함할 수 있는 승인·검수 절차
13. 재채점 추천이 우선 최적화할 목표: 오류 발견률, 비용, 교사 검토 시간
14. 교사별 보정과 과목 공통 보정의 분리 범위
15. 적응형 알고리즘 도입을 허용할 최소 검증 조건

권장 원칙은 “안전한 기본값을 제공하되 교사가 이해할 수 있는 범위에서 변경 가능”이다.

## 32. 완료 정의

차기 버전의 핵심 개발이 완료됐다고 판단하려면 다음 조건을 모두 만족해야 한다.

### 기능

- 보고서·수행평가와 정기고사 서술형 대표 흐름이 처음부터 끝까지 동작한다.
- 공식 루브릭이 있는 경우와 없는 경우를 모두 처리한다.
- 통합 스캔을 학생별로 분할하고 채점에 연결한다.
- 소문항별 점수와 대문항·총점 합계를 정확히 계산한다.
- 여러 회차와 선택 추가 회차가 동작한다.
- AI 점수, 표준편차, 수동 점수, 차이를 확인한다.
- 교사가 점수를 수정·승인·확정한다.
- 기존 프로젝트와 결과를 읽는다.

### 안정성

- 핵심 자동 테스트가 모두 통과한다.
- 브라우저 핵심 흐름 테스트가 통과한다.
- 중지·오류 후 성공 결과를 잃지 않는다.
- 공급자·모델·기준 버전이 회차에 정확히 기록된다.
- 구형 데이터 마이그레이션 전후 결과가 보존된다.

### 사용성

- 첫 사용자가 프로젝트 유형을 구분할 수 있다.
- 각 화면에서 다음 할 일을 알 수 있다.
- 비활성 버튼의 이유를 알 수 있다.
- 통합 분할 파일의 저장 위치를 찾을 수 있다.
- 오류 발생 후 재개 방법을 알 수 있다.

### 검증

- 표준 가상 자료로 보고서와 서술형 전체 흐름을 검증한다.
- 실제 학교 자료의 소규모 표본으로 교사가 직접 검증한다.
- AI 점수와 교사 수동 점수 비교 보고서를 확인한다.
- 채점 외 추가 입력 없이 개발용 성능자료 묶음을 생성한다.
- 피드백 묶음에 실제 학생 식별정보, 대응표, 인증 정보, 원본 자료가 없는지 확인한다.
- 치명적 오류가 해결된 뒤에만 배포 빌드를 만든다.
